# Desktop UI

Первый desktop UI построен на PySide6 и использует backend-ядро через
`ApplicationContainer` и `AgentApplicationService`.

## Запуск

```bash
python scripts/run_desktop_app.py
```

Конфигурация берётся из переменных окружения `AppConfig`, как и CLI. Сборка
container не вызывает COM и не читает Outlook. В режиме `outlook_readonly`
Outlook может быть затронут только при явном запуске агента.

## Страницы

- **Агенты**: таблица сохранённых агентов и карточка выбранного агента.
- **Создать агента**: запрос, preview `AgentSpec`, сохранение агента.
- **Запуски**: список запусков агента, карточка выбранного запуска и запуск агента.
- **Состояние запуска**: подробный `AgentRuntimeState` по `run_id`.
- **События**: таблица событий запуска, фильтр по `event_type`, details-панель.
- **Подтверждения**: pending approvals, карточка запроса, approve/reject.
- **Настройки**: локальный JSON редактор `AppConfig`.
- **Журнал**: простая панель сообщений UI.

## Создание агента

1. Откройте страницу **Создать агента**.
2. Введите текстовый запрос.
3. Нажмите **Предпросмотр**.
4. Проверьте вкладки:
   - `Общее`
   - `Данные`
   - `Инструменты`
   - `Граф`
   - `JSON`
5. Нажмите **Сохранить**.

Preview вызывает `AgentApplicationService.build_preview(...)`; агент не
сохраняется и не запускается до явного действия пользователя.

## Запуск агента

На странице **Агенты** можно выбрать агента и нажать
**Запустить выбранного агента**. На странице **Запуски** можно ввести `agent_id`
и нажать **Запустить агента**. В обоих случаях UI вызывает только
`AgentApplicationService.run_agent(...)`.

Если backend вернул ошибку внешней интеграции, UI показывает её в сообщении и в
карточке запуска, а не скрывает.

## Фоновый запуск агента

Runtime нельзя запускать в UI-потоке: в режиме `outlook_readonly` read-only
COM-вызов Outlook выполняется через subprocess и может отвечать долго до
timeout. Чтобы окно не зависало, кнопки запуска используют `AgentRunWorker` и
`AgentRunThreadManager`.

`AgentRunWorker` — это `QObject`, который перемещается в `QThread` через
`moveToThread`. Worker вызывает только `AgentApplicationService.run_agent(...)`
или `AgentApplicationService.run_agent_spec(...)` и отдаёт результат сигналами.
Он не наследуется от `QThread` и не импортирует COM, ToolGateway или Runtime.

`AgentRunThreadManager` управляет одним активным запуском. На MVP параллельный
второй запуск отклоняется, чтобы не усложнять состояние UI. После завершения
поток закрывается, кнопка запуска разблокируется, а список запусков обновляется.

COM по-прежнему вызывается только backend-слоем через `SubprocessComWorker`.
Если Outlook завис или не ответил, backend возвращает `WORKER_TIMEOUT`, а UI
показывает понятное сообщение: проверьте Outlook, профиль и модальные окна.

## События

Откройте страницу **События**, введите `run_id` и нажмите **Показать события**.
Можно фильтровать таблицу по `event_type`. Полные `details` выбранного события
показываются в панели справа, а не в таблице.

## Подтверждения

Откройте страницу **Подтверждения** и нажмите **Обновить**. Выберите pending
approval, при необходимости заполните комментарий и нажмите **Подтвердить** или
**Отклонить**. UI вызывает
`AgentApplicationService.resume_run_by_approval(...)`.

Подтверждение не обходит safe mode. Если подтверждённое действие связано с
`email.send`, backend всё равно блокирует реальную отправку через
`SEND_DISABLED_FOR_SAFETY`; UI показывает понятное предупреждение.

## Автообновление

На страницах **Запуски**, **События** и **Подтверждения** есть checkbox
**Автообновление**. Интервал по умолчанию — 3000 мс. Автообновление только
читает SQLite через service-слой и не запускает агентов.

## Настройки

Страница **Настройки** сохраняет параметры приложения в локальный JSON-файл:

```text
./data/settings.json
```

Доступные режимы:

- `fake`: используются fake tools, Outlook не нужен.
- `outlook_readonly`: доступны read-only Outlook-инструменты через subprocess.
- `offline`: инструменты внешних интеграций не используются.

На странице можно настроить путь к SQLite, LLM Planner, параметры COM worker и
лимиты чтения Outlook. После нажатия **Сохранить** нужен перезапуск приложения:
`ApplicationContainer` уже собран, а Runtime, workers и storage на лету пока не
пересобираются.

Кнопка **Проверить настройки** делает только локальную валидацию полей. Она не
вызывает Outlook, не создаёт COM worker и не делает HTTP-запрос к LLM.

## Что UI не делает

- Не вызывает COM напрямую.
- Не отправляет письма.
- Не создаёт черновики.
- Не меняет Outlook.
- Не поднимает web-сервер.
- Не использует Redis или Celery.
- Не собирает `Runtime`, `ToolGateway` или `ToolRegistry` вручную.

## Почему через AgentApplicationService

UI не содержит бизнес-логику и не собирает `Runtime`, `ToolGateway`,
`ToolRegistry` или workers вручную. Виджеты обращаются к
`container.agent_service`, а service уже знает, как работать с агентами,
запусками, событиями и подтверждениями. Это сохраняет UI тонким и позволяет в
будущем заменить экран или добавить новые сценарии без дублирования backend
правил.
